вЂ‹вЂ‹Category 1: High-Performance Computing (HPC) & Grid Workload ArchivesвЂ‹вЂ‹These are excellent sources as they explicitly deal with jobs submitted to shared computing resources.

вЂ‹вЂ‹Grid Workload Archive (GWA) Traces:вЂ‹вЂ‹

вЂ‹вЂ‹Specific Traces:вЂ‹вЂ‹ GWA-T-4 (AuverGrid), GWA-T-12 (Nordic Grid), GWA-T-1 (LHCb), etc.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Submission time, wait time, runtime (execution time), number of CPUs used, user ID, requested resources.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹
arrival_time: Submission time.
burst_time: Runtime.
priority: Can be derived (e.g., shorter jobs might implicitly have higher priority in some systems, or policy-based priorities could be inferred from user/group IDs).
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Historically at gwa.ewi.tudelft.nl and various academic sites. Search "Grid Workload Archive" and specific trace names.
вЂ‹вЂ‹Parallel Workloads Archive (PWA):вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ A broader collection of logs from various parallel supercomputers and clusters (e.g., from Los Alamos, CTC, KTH).
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Similar to GWA вЂ“ submit time, wait time, run time, requested processors, memory, user/group info.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Similar to GWA.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ www.cs.huji.ac.il/labs/parallel/workload/ (a primary source for many PWA traces).
вЂ‹вЂ‹HPC Workload Traces (Specific Institutions):вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ Logs from individual supercomputing centers not always aggregated into GWA/PWA.
вЂ‹вЂ‹Examples:вЂ‹вЂ‹ LANL (Los Alamos National Lab) traces, LLNL (Lawrence Livermore National Lab) traces.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Job submission time, execution time, allocated resources (nodes/CPUs), queue names (can imply priority).
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Straightforward for arrival/burst. Priority might be inferred from queue assignment or explicit job priority flags if available.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Often found on the websites of the supercomputing centers themselves or linked through research papers. Search "LANL workload trace," "LLNL job logs."
вЂ‹вЂ‹Category 2: Cloud Computing & Virtual Machine WorkloadsвЂ‹вЂ‹These provide insight into diverse workloads running on shared infrastructure, often including heterogeneous tasks.

вЂ‹вЂ‹Google Cluster Traces:вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ Logs from large Google clusters. Although older, they are seminal for understanding cloud workloads.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Task submission time, start time, end time, requested CPU/memory, actual CPU/memory usage, job type.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹
arrival_time: Task submission time.
burst_time: (End time - start time) or aggregated CPU usage.
priority: Available in some versions, or inferable from job/task type.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Look for "Google Cluster Trace v2" or later versions. Often hosted by institutions like Google itself or related research groups.
вЂ‹вЂ‹Microsoft Azure Traces:вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ More recent and massive traces from Microsoft's cloud infrastructure.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ VM creation/deletion times, resource utilization (CPU, memory), job types.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Can be used to model VM lifetimes as "bursts" or derive individual task runtimes. Requires more processing to map to discrete CPU jobs.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Search "Microsoft Azure public dataset" or "Azure trace data."
вЂ‹вЂ‹Alibaba Cluster Traces:вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ Traces from Alibaba's production clusters.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Job/task submission, resource requests, actual usage, start/end times.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Similar to Google traces.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Search "Alibaba Cluster Trace." Often provided by Alibaba Cloud for research purposes.
вЂ‹вЂ‹Category 3: Web Server & Network Logs (Requires More Processing)вЂ‹вЂ‹These are less direct for "CPU scheduling" but offer real-world event streams from which "tasks" and "service times" can be derived.

вЂ‹вЂ‹WorldCup98 Web Server Log:вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ Famous log of all requests to the World Cup 1998 website.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Request arrival time, URL requested, response size.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹
arrival_time: Time of request.
burst_time: Can be modeled as the server processing time for a request (requires additional modeling or small-scale measurement).
priority: Could be assigned based on request type (e.g., critical pages).
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Search "WorldCup98 web log" or "Internet Traffic Archive."
вЂ‹вЂ‹NASA Kennedy Space Center Web Server Logs:вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ Logs of requests to the NASA KSC website.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Similar to WorldCup98 logs.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Similar to WorldCup98.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Also often found in "Internet Traffic Archive" or related academic network research repositories.
вЂ‹вЂ‹Category 4: Operating System/User Activity Logs (Less Common Publicly, but Possible)вЂ‹вЂ‹These are harder to come by publicly due to privacy, but conceptually offer direct OS-level activity.

вЂ‹вЂ‹OS-level Process Traces (e.g., from Linux perf or sysstat):вЂ‹вЂ‹
вЂ‹вЂ‹Description:вЂ‹вЂ‹ Detailed logs of process creation, execution, and termination events directly from an OS. Not usually archived for public general use due to system-specific nature and privacy.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Process ID, creation time, CPU usage, context switches, I/O.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Direct arrival_time (creation), burst_time (CPU usage), priority (actual OS priority if logged).
вЂ‹вЂ‹Links:вЂ‹вЂ‹ You would typically generate your own by running tools like perf or strace on a test system, rather than downloading an existing large-scale public archive. This counts as real-world data generation but not a "downloadable dataset."
вЂ‹вЂ‹Category 5: University/Institutional Cluster LogsвЂ‹вЂ‹Similar to HPC, but often from diverse academic environments.

вЂ‹вЂ‹University of Arizona Condor Logs:вЂ‹вЂ‹
вЂ‹вЂ‹Description:вЂ‹вЂ‹ Logs from a Condor batch system cluster.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Job submission, start, end times, resource usage, user/group.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Similar to HPC traces.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Often found through "Condor Workload" searches or academic research archives.
вЂ‹вЂ‹Category 6: Synthetic Benchmarks (providing "real" execution data)вЂ‹вЂ‹While the raw "dataset" is synthetic (the benchmark itself), the execution results are real-world measurements.

вЂ‹вЂ‹SPEC CPU 2017/2006/2000 Output Logs:вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ While the benchmark itself isn't a "trace," running the benchmarks generates real-world CPU usage and completion times. If you can find logs of actual SPEC runs from published research, these count.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Execution time for specific workloads, CPU cycles, memory access patterns.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ You would use these execution times as burst_time. You'd then need to simulated arrival patterns and priorities for multiple processes based on these.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ The SPEC website provides the benchmarks. Finding output logs from various systems requires searching academic papers that publish their SPEC results.
вЂ‹вЂ‹TPC Benchmarks (e.g., TPC-C, TPC-H):вЂ‹вЂ‹

вЂ‹вЂ‹Description:вЂ‹вЂ‹ Industry-standard benchmarks for transactional and analytical database systems. They stress CPU, I/O, and memory.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ Transaction rates, response times under load.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Very indirect. You'd model individual transactions as "tasks" and their processing time as burst_time. This is more about server workload than individual CPU task scheduling.
вЂ‹вЂ‹Links:вЂ‹вЂ‹ www.tpc.org
вЂ‹вЂ‹Category 7: Mobile/Edge Computing TracesвЂ‹вЂ‹Increasingly relevant for distributed scheduling.

вЂ‹вЂ‹Mobile Workload Traces (e.g., from smartphone usage):вЂ‹вЂ‹
вЂ‹вЂ‹Description:вЂ‹вЂ‹ Logs of application usage, CPU consumption, and network activity on mobile devices.
вЂ‹вЂ‹Data Provided:вЂ‹вЂ‹ App launch/close times, CPU cycles used by apps, battery consumption.
вЂ‹вЂ‹Derivation:вЂ‹вЂ‹ Can define app execution as a "job" with arrival_time (launch) and burst_time (total CPU cycles used).
вЂ‹вЂ‹Links:вЂ‹вЂ‹ Often found in mobile computing research papers or dedicated project websites.
вЂ‹вЂ‹Advice for using these datasets:вЂ‹вЂ‹

вЂ‹вЂ‹Data Cleaning and Preprocessing:вЂ‹вЂ‹ Expect to spend significant time cleaning, parsing, and transforming these raw logs into a format suitable for your scheduling simulations (i.e., (arrival_time, burst_time, priority) for each job/process).
вЂ‹вЂ‹Deriving Priority:вЂ‹вЂ‹ This is often the trickiest part for real-world traces. You might have to infer it from user/group IDs, job classes, requested queue, or by making reasonable assumptions (e.g., jobs submitted by a specific research group have higher priority). Clearly state your assumptions in your research.
вЂ‹вЂ‹Scalability:вЂ‹вЂ‹ Many of these traces are huge. Ensure you have the computational resources and scripting skills to handle them.
вЂ‹вЂ‹Documentation:вЂ‹вЂ‹ Always refer to the original documentation of the trace for precise column meanings and units.
By leveraging a diverse selection from these categories, you can build a robust analysis grounded in various real-world computing scenarios.
